================================================================================
  DESIGN OVERVIEW — ラクユーZ工法協会 Next.js Site
================================================================================

This document describes the visual design system, component patterns, layout
conventions, and architectural decisions used throughout the site.

────────────────────────────────────────────────────────────────────────────────
1. BRAND & VISUAL LANGUAGE
────────────────────────────────────────────────────────────────────────────────

  The site communicates authority, technical precision, and reliability —
  qualities expected of a civil engineering association with 30+ years of
  industry experience and a national award (インフラメンテナンス大賞 優秀賞).

  Visual direction:
    • Corporate navy as the dominant dark tone — conveys professionalism
    • Professional blue accent — active, technological, trustworthy
    • Muted gold — reserved exclusively for the award badge in the hero
    • Clean white backgrounds with light slate borders for content sections
    • Photography-forward: full-bleed images with dark overlay on hero/page headers

7. ANIMATION SYSTEM  (src/lib/animations.ts + Framer Motion)
────────────────────────────────────────────────────────────────────────────────

  NAMED VARIANTS
  ───────────────
  fadeUpVariants        opacity 0→1, y 20→0, duration 0.6
  staggerContainerVariants   staggerChildren: 0.1, delayChildren: 0.2
  staggerItemVariants   opacity 0→1, y 16→0, duration 0.5
  pageVariants          page-level fade + slight y on enter/exit

  USAGE PATTERN
  ──────────────
  • Hero content: direct animate (not scroll-triggered), delays 0.1–0.4s
  • Section content: whileInView + viewport={{ once: true }}
  • Cards in grids: staggerContainerVariants wrapping staggerItemVariants
  • Modals: scale 0.95→1 + y 12→0, duration 0.2 (fast, feels snappy)
  • Backdrop: opacity only, duration 0.2

  CUSTOM HOOKS  (src/hooks/useAnimations.ts)
  ────────────────────────────────────────────
  useHeaderScroll(threshold)   Returns boolean — true when scrollY > threshold
  useCountUp(target, duration, start, trigger)   Animated number counter for stats

  REDUCED MOTION
  ───────────────
  globals.css includes @media (prefers-reduced-motion: reduce) that collapses
  all animation/transition durations to 0.01ms system-wide.

8. RESPONSIVE BREAKPOINTS
────────────────────────────────────────────────────────────────────────────────

  Tailwind defaults (used throughout):
    sm:   640px   Stacked → side-by-side for small elements
    md:   768px   Single column → 2-column grids, larger hero padding
    lg:   1024px  2-column → 3-column grids, desktop navigation appears
    xl:   1280px  Max layout width effectively reached

  KEY RESPONSIVE BEHAVIORS
  ─────────────────────────
  Navigation:   lg:flex hidden / lg:hidden flex (hamburger)
  Mobile menu:  Full-screen overlay (fixed inset-0), bg-primary-950
  Hero text:    text-center on mobile → lg:text-left
  CTA buttons:  flex-col → md:flex-row
  Info tables:  flex-col rows → sm:flex-row
  Filter bars:  flex-col → sm:flex-row, flex-wrap
  Touch targets: All interactive elements min-h-[44px] (iOS HIG compliance)

9. IMAGERY CONVENTIONS
────────────────────────────────────────────────────────────────────────────────

  All images use Next.js <Image> component with:
    fill + object-cover   — Full-bleed hero/page-header photos
    fill + object-contain — Construction photos, equipment images (preserve aspect)
    sizes prop            — Always provided for LCP optimization
    priority              — Set on above-the-fold images (hero, page hero)
    unoptimized           — Set on complex photography to bypass Next.js compression
    quality={100}         — Hero backgrounds for best fidelity

10. WORDPRESS CMS INTEGRATION ARCHITECTURE
────────────────────────────────────────────────────────────────────────────────

  FLOW
  ──────
  WordPress.com CMS  →  REST API  →  src/lib/wordpress.ts  →  Page components

  FALLBACK BEHAVIOR
  ──────────────────
  Every component that fetches WordPress data has a fallback:
    1. isWpConfigured() returns false   → renders MOCK_NEWS_POSTS / MOCK_DOCUMENTS
    2. fetch() throws or returns !ok    → catches and renders mock data silently
  This ensures the site always renders something, never breaks.

  The "preview" label (tiny italic) appears in the WpNewsPanel when mock data
  is being shown instead of live data.

  DATA CACHE STRATEGY  (Next.js fetch revalidation)
  ────────────────────────────────────────────────────
  Posts list:       revalidate: 300   (5 minutes)
  Category list:    revalidate: 3600  (1 hour)
  Documents:        revalidate: 600   (10 minutes)

  CLIENT-SIDE FETCHING
  ─────────────────────
  All WordPress data is fetched client-side via useEffect (all pages are
  "use client"). Data is not server-rendered. This was chosen to keep the
  architecture simple and avoid ISR/SSR cache complexity with WordPress.com.
